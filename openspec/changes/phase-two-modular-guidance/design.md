## Context

当前 `dl-apply` 将根目录的 `DESIGN.md`、`STYLE_PREVIEW.md` 和一个 preset 拼成单个受管区块，写入目标项目的 `AGENTS.md`。现有调用形式是 `dl-apply <目录> [preset]` 与 `dl-apply --check <目录> [preset]`，默认 preset 为 `dark`。`--check` 目前只确认起始标记存在，不能发现 preset 错误、上游内容过期、结束标记缺失或区块被手工改写。

第二阶段需要把所有项目都要读取的核心契约与专项指导分开。核心仍包含共享 DNA、NEVERS、三拨盘、规则优先级、页面级功能契约和最小交付门禁。四个专项模块只在调用者显式选择时注入，模块 ID 固定为 `ui-patterns`、`motion`、`data-vis`、`preferences`。`STYLE_PREVIEW.md` 仍属于默认受管内容，不是可选模块。

仓库同时维护根目录规范和 `starter/.ai/` 镜像。对应文件必须保持 byte-for-byte 一致，CI 以 `cmp` 验证。实现只使用现有 Bash 与标准 Unix 工具，不引入包、服务或后台进程，也不触碰目标项目的 CSS、HTML、字体和应用源码。

## Goals / Non-Goals

**Goals:**

- 保持所有旧调用及其默认 preset、默认 `STYLE_PREVIEW.md` 注入行为兼容。
- 提供四个显式启用的模块，并以唯一的规范顺序生成稳定受管区块。
- 让 apply 与 check 使用同一个期望区块生成过程，使 preset、模块集合、模块顺序和全部上游字节都可严格验证。
- 对完整受管区块执行幂等替换，同时保留区块外全部字节和目标文件原有权限。
- 在标记损坏或参数非法时拒绝写入，并用稳定退出码区分匹配、不匹配和用法或安全错误。
- 让根目录规范与 `starter/.ai/` 对应镜像保持字节一致，并由无外部依赖的测试和 CI 保护。

**Non-Goals:**

- 不实现预览服务器、文件监听器、后台进程或任何网络服务。
- 不增加外部运行时或测试依赖。
- 不自动修改目标项目的 CSS、HTML、字体、token、组件或其他应用源码。
- 不把四个模块默认注入所有项目，也不根据项目内容自动猜测模块。
- 不迁移或复制任何个人品牌、个人偏好或与本仓库无关的身份内容。

## Decisions

### 1. 核心契约与专项模块分层

`DESIGN.md` 继续是默认核心契约，并补齐页面级功能约束。`STYLE_PREVIEW.md` 继续无条件位于核心与 preset 之间。四个模块各自保存在独立参考文件中，只有 `--modules` 显式选择后才进入受管区块。

`preferences` 模块只注入治理说明和空模板；实际项目偏好写入目标项目根目录的 `DESIGN_PREFERENCES.md`，该文件不属于 `dl-apply` 受管区块，刷新 preset、模块或上游规范时不得覆盖它。

选择这一方案是为了控制默认上下文体积，同时保留当前风格试衣间行为。备选方案是把全部内容继续追加到 `DESIGN.md`，但会让每个项目承担无关 prompt。另一个备选方案是把 `STYLE_PREVIEW.md` 也改成模块，这会改变现有默认注入，因而拒绝。

### 2. 模块注册表是唯一事实来源

`scripts/dl-apply-registry.sh` 维护固定注册表，规范顺序为 `ui-patterns`、`motion`、`data-vis`、`preferences`。注册表同时定义合法 ID、源文件路径和输出顺序；展示标题直接使用规范 ID。`--modules` 接受逗号分隔 ID，解析时去除重复项，但不采用用户输入顺序，最终始终按注册表顺序输出。空模块集合等同于未传 `--modules`。

固定顺序使同一集合只有一种字节表示，简化幂等和严格检查。保留输入顺序虽然更直观，但会让相同集合产生多种区块，增加无意义漂移。自动发现 `references/` 文件也被拒绝，因为文件名变化会悄然改变 CLI 契约。

### 3. 兼容式参数语法

旧形式保持有效：`dl-apply <目录> [preset]` 和 `dl-apply --check <目录> [preset]`。新增 `--modules <逗号分隔列表>`，可与 apply 或 check 组合。未提供 preset 时仍使用 `dark`，未提供模块时不注入任何可选模块，`STYLE_PREVIEW.md` 仍照常注入。

未知选项、缺失参数、未知 preset、未知模块 ID 或不可用源文件属于调用或配置错误。不会通过位置参数重解释这些错误，以免合法拼写错误造成静默写入。

### 4. 单一规范化区块生成器

apply 与 check 先从同一组输入生成期望区块：起始标记、当前 preset 标题、固定说明、`DESIGN.md`、`STYLE_PREVIEW.md`、所选模块和 preset，段间分隔及尾随换行也完全一致。模块按注册表顺序插入，并在区块元数据中记录规范化后的模块集合，使 check 能报告期望配置。

复用同一生成路径可以避免 apply 和 check 各自拼接导致规则漂移。只比较摘要或只搜索标题的方案无法发现上游内容被编辑，因此不采用。

### 5. 严格 check 与退出码

`--check` 不写文件。它先验证标记结构，再生成期望区块并与目标中的唯一受管区块逐字节比较。只有 preset、规范化模块集合、模块顺序和全部上游内容完全一致时返回 `0`。目标文件或区块缺失、完整区块存在但内容与期望不同，返回 `2`。参数非法、源文件不可读、出现重复标记、只有单边标记、结束标记早于起始标记等无法安全解释的情况返回 `1`。

退出码 `2` 专用于可通过正常 apply 修复的“不匹配”，`1` 表示调用错误或需人工处理的损坏状态。这样 CI 可以区分“需要刷新”与“不能安全继续”。

### 6. 安全替换、幂等与文件保持

apply 只接受两种目标状态：完全没有标记，或恰好一个顺序正确的完整标记对。没有标记时，将受管区块追加到现有内容后；已有完整区块时，只替换该区块。任何畸形或重复标记都在写临时文件前拒绝，返回 `1`，目标文件保持不变。

缩进、尾随空格或轻微改写但仍呈现为 design-language HTML 标记的近似行也属于畸形标记，不得按“无标记”追加新区块。

替换过程以原文件为字节输入，复制起始标记前和结束标记后的内容，不对区块外空行、行尾、编码或文本做规范化。结果通过同目录临时文件生成，并在替换前读取原 Unix mode bits，替换后恢复同一 mode bits。连续两次以相同参数 apply，第二次不得改变文件字节或 mode bits。

该 Bash CLI 的安全边界是“当前用户拥有的可信项目目录”：不得通过 `sudo` 修改其他用户文件，也不承诺保留 ACL、xattr、SELinux label 等扩展元数据。脚本在扫描后和替换前复核目标 device/inode，降低检查后被替换的风险；不可信共享目录不在支持范围内。

直接用 `awk` 重写整份文件可能改变区块外字节或权限，因此实现需把“受管区块定位”和“原始字节拼接”分开。原地编辑也可能在中断时留下半写文件，不采用。

### 7. 根目录与 starter 镜像契约

每个核心、preset 和模块文件在根目录与 `starter/.ai/` 都有对应副本，内容必须 byte-for-byte 相等。测试和 CI 使用 `cmp` 检查全部映射，不通过文本解析或语义比较放宽差异。

字节相等能保证 starter 与直接运行 `dl-apply` 获得同一指导内容。只维护符号链接不适合复制、打包和跨平台使用，因此保留实体镜像并由 CI 防漂移。

## Risks / Trade-offs

- [风险] 严格字节比较会把无害的换行变化也判为不匹配。→ 缓解：apply 与 check 共用规范化生成器，CI 用同一字节标准，差异可通过重跑 apply 修复。
- [风险] Bash 参数解析和字节保持实现比当前脚本复杂。→ 缓解：限制为四个固定模块和单一 `--modules` 语法，并用无外部依赖的行为测试覆盖旧调用、重复模块、顺序、退出码、权限与区块外字节。
- [风险] 拒绝畸形标记会要求人工清理，不能自动恢复。→ 缓解：错误信息明确指出标记结构问题，且在任何写入前失败，优先保护用户文件。
- [取舍] 固定模块顺序不保留用户输入顺序。→ 缓解：文档公开规范顺序；确定性输出换来可靠幂等和简单审计。
- [取舍] 模块不自动选择，调用者需要了解四个 ID。→ 缓解：CLI 帮助和文档列出 ID 与用途；显式选择避免错误猜测和默认 prompt 膨胀。
