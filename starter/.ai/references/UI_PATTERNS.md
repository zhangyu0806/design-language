# UI_PATTERNS.md：功能型 UI 决策参考

> 可选模块 `ui-patterns`。用于复杂组件、任务流程和状态恢复。它补充 `DESIGN.md`，不替代页面级 Functional Contract、NEVERS、preset 或语义 token 纪律。

## 1. 组件决策档案

不要先问“该用哪种卡片”，先为重要组件写一份简短档案：

```text
任务：用户要完成什么，而不是组件叫什么
入口：用户从哪里、带着什么上下文到达
信息：做决定前必须看到什么
主动作：当前唯一主动作及完成信号
次动作：可以推迟或退出的动作
状态：初始 / 进行中 / 成功 / 空 / 错误 / 禁用
边界：权限、数据、设备、网络或不可逆限制
恢复：失败后如何保留输入、返回安全位置或重试
证据：为何选择此模式，参考了什么真实约束
```

选组件时比较认知成本、错误代价、频率和空间，而不是照搬流行样式：

- 高频、低风险、结果立即可见的动作可直接执行。
- 低频或高风险动作要解释后果，并给撤销或明确确认。
- 选项互斥且数量少时优先直接展示；数量多、需搜索或依赖上下文时再用选择器。
- 信息要比较时并列展示；信息要逐步承诺时分步展示。不要为了“干净”隐藏决策所需内容。
- 同一任务优先沿用已有模式。新模式必须解决旧模式无法解决的明确问题。

## 2. 完整任务流

复杂流程至少检查以下八段。简单页面可以合并阶段，但不能漏掉真实会发生的状态。

| 阶段 | 用户问题 | 设计责任 |
|---|---|---|
| Entry | 我为什么来到这里？ | 说明位置、价值、前置条件和主任务 |
| Setup | 开始前要准备什么？ | 收集最少必要信息，给默认值和约束 |
| Act | 我现在该做什么？ | 主动作清楚，次动作不抢层级 |
| Progress | 系统正在做什么？ | 显示阶段、等待原因、可否取消或离开 |
| Result | 完成了吗，结果是什么？ | 明确成功、产物、影响范围和下一步 |
| Exception | 哪里出了问题？ | 用人能理解的语言指出问题和影响 |
| Recovery | 怎么继续？ | 保留有效输入，提供重试、修正或替代路径 |
| Exit | 如何安全离开？ | 说明保存状态、未完成影响和返回位置 |

`Recovery-Exit` 要一起审：恢复失败时仍应能安全退出，退出后也应知道如何回来。不要把错误页做成死路。

## 3. 状态与反馈

- Loading 表达正在等待什么，长任务要给阶段或进度依据；不要用孤立 spinner 代替信息。
- Empty 说明为何为空，并提供能产生第一条内容的动作；“暂无数据”不是完整指引。
- Error 区分用户可修正、可重试和需要外部处理的情况，不把内部错误码当正文。
- Disabled 只有在原因可理解时使用。若用户需要知道为何不能操作，就在附近解释。
- Success 不只变绿，要确认完成对象、保存位置、影响范围和后续动作。
- 乐观更新只用于失败可回滚且用户不会误判高风险结果的操作。

## 4. 可访问性与键盘路径

- 视觉顺序、DOM 顺序和键盘顺序保持一致。
- 对话框打开后管理初始焦点，Esc 可关闭时必须可用，关闭后焦点回到触发点。
- 动态结果用合适的 `aria-live` 语义播报，但避免重复朗读整个区域。
- 标签、错误和帮助信息建立程序化关联；placeholder 不能代替 label。
- 颜色不作为唯一状态信号，图标必须配文本或可访问名称。

## 5. 审核

- [ ] 组件决策档案说明了任务、主动作、状态、边界、恢复和证据
- [ ] Entry / Setup / Act / Progress / Result 路径连续
- [ ] Exception / Recovery / Exit 没有死路，输入和上下文尽量保留
- [ ] 高风险动作的确认、撤销与结果反馈匹配真实代价
- [ ] 鼠标、键盘、读屏和窄屏都能完成同一主任务
